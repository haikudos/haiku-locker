# Signer Sidekick — standalone dApps

Five single-file HTML dApps that walk the **on-chain / wallet steps** of bringing a Stacks PoX-5 signer online, plus one offline key helper. Each file is self-contained: open it in a browser, no build step, no server. They reuse the exact contract interfaces, chain IDs, and pre-sign verification from [`stx-labs/signer-sidekick`](https://github.com/stx-labs/signer-sidekick) rather than reinventing them.

These do **not** install the node or signer, and they never touch the signer's private key beyond the offline helper. They talk to your wallet over SIP-030 (`@stacks/connect` 8.2.6) and to the public Hiro API for reads.

## What's in scope vs not

Out of scope (upstream, before these tools): running the Stacks node and the signer process. See the [node guide](https://docs.stacks.co/operate/run-a-node) and [signer guide](https://docs.stacks.co/operate/run-a-signer).

In scope: everything after the node + signer are running — key material, funding, deploy, register, verify.

## Two separate keys — keep them apart

This is deliberate and the whole reason for the split-wallet design:

- **Signer key** — a secp256k1 key that lives on the signer node. It signs blocks, so it sits on an internet-facing box and is inherently more exposed. Generated by dApp 2 (or, preferred, on the node host). The chain only ever needs its **public** key.
- **Manager-admin / staking wallet** — the funded key that deploys the manager and broadcasts `register-self`. Keep this in a hardware wallet (Ledger) on mainnet. This is the key you connect in dApps 1, 3, and 4.

Never put the signer private key in a wallet, and never paste it into dApps 1/3/4/5 or anywhere online. They only need the public key.

## Order of operations

| Step | Tool | Key used | Signs? |
|---|---|---|---|
| 0 | **01 Connect + Preflight** | manager-admin | no (reads) |
| 1 | **02 Signer key helper** (or node-host CLI) | signer | no (offline gen) |
| 2 | *On signer host:* configure signer with the key | signer | — |
| 3 | **03 Deploy signer-manager** | manager-admin | yes (deploy) |
| 4 | *On signer host:* `stacks-signer generate-staking-signature --signer-manager <manager> --auth-id <id> --json` | signer | produces grant JSON |
| 5 | **04 Register-self** (paste grant JSON) | manager-admin | yes (contract call) |
| 6 | **05 Rewards + status** | none | no (reads) |

The contract source for step 3 comes from Sidekick's `manager render` output (pinned, hash-verified). These dApps deploy/broadcast exactly what you give them and show the hash/args first; they do not generate contract source.

## Running them

1. Install Leather. Add a **custom network** with key `pox5-testnet` and API URL `https://api.testnet-pox5.hiro.so` (Xverse is not enabled for this custom network). The wallet must report your `ST…` admin address.
2. Open any `.html` file directly in the browser. They load `@stacks/connect@8.2.6`, `@stacks/transactions@7.5.0`, and `@noble/secp256k1@2.1.0` from esm.sh on first use — the machine needs internet for that (except dApp 2, which you should run offline after it loads once).
3. Fund the admin address from the testnet faucet before deploying/registering.

Ordinary Stacks testnet is a **different chain** (`0x80000000`). This dedicated PoX-5 testnet is `0x80000005`. Mainnet is `0x00000001`. The dApps pin the expected PoX-5 boot contract (`ST000000000000000000002AMW42H.pox-5` on testnet) and warn if the live network disagrees.

## Security tier

Per the `stacks-dapp-architecture` posture:

- **On PoX-5 testnet these run at Tier 1** — faucet tokens, no real value. Financial safeguards (slippage, price sanity, confirmation friction) are not applicable and are deferred.
- **The structural invariants are on at every tier**, so nothing has to be retrofitted for mainnet:
  - post-condition mode is **deny** on every asset-capable call (deploy and register-self move no assets, so 0 post-conditions);
  - contract identity is **pinned** — the manager principal you enter and the PoX-5 boot contract are checked against the live network before signing;
  - **keys stay in the wallet** — the dApps build and display requests; the wallet signs.
- **Mainnet is Tier 2.** The same code runs there, but real value can move, so the promotion checklist below becomes mandatory.

### How this differs from full Sidekick

Sidekick's backend "seals" each transaction into an integrity-checked intent (manifest hash + review fields) and the browser only executes that sealed intent. These standalone dApps drop the backend: they reconstruct the call client-side and pin/verify contract identity in the page instead. dApp 4 replicates Sidekick's full pre-sign verification (live message-hash recovery, signature validity, grant-not-used) so the safety checks are equivalent — but the trust root is the pinned value you enter, not a backend seal. For unattended or multi-operator mainnet use, run Sidekick itself.

## Mainnet promotion checklist (Tier 1 → Tier 2)

Crossing to mainnet is an explicit promotion, not flipping the network dropdown. Before pointing any of these at mainnet:

- [ ] Switch the network selector to **Mainnet** deliberately, and confirm the wallet is a mainnet (`SP…`) account on a Ledger.
- [ ] Re-pin contract identity: confirm the mainnet PoX-5 boot contract and set the expected value in the code (currently `null`/warn-only for mainnet in dApps 4 and 5 — replace with the verified mainnet `.pox-5` principal).
- [ ] Confirm the manager contract source SHA-256 (dApp 3) matches the mainnet deployment manifest from `sidekick manager render`.
- [ ] Confirm post-condition mode is **deny** on every call (already enforced) and that any future asset-moving call carries explicit-amount post-conditions.
- [ ] Remove every testnet shortcut: faucet assumptions, the offline-key acknowledgement bypass, any hardcoded testnet address.
- [ ] Add financial safeguards for any value-moving action you add later (confirmation friction, sanity bounds).
- [ ] Verify the signer key was generated and stored per the signer guide, isolated from the Ledger-backed admin key.
- [ ] Name an owner for contract-level risk (audit status, oracle trust, front-running/MEV, governance) — out of scope for these frontends but real at mainnet.

Until the mainnet PoX-5 principal is pinned in the code, dApps 4 and 5 only **warn** instead of blocking on an unexpected mainnet contract. Pin it before mainnet use.

## Files

- `01-connect-preflight.html` — connect admin wallet, verify chain/address, balance, PoX-active, readiness checks.
- `02-signer-key-helper.html` — offline secp256k1 signer keygen with guardrails (prefer node-host CLI).
- `03-deploy-manager.html` — deploy the reviewed `signer-manager` contract (`clarityVersion 6`, deny mode).
- `04-register-self.html` — verify the signer grant against live PoX-5, then broadcast `register-self`.
- `05-rewards-status.html` — read-only registration / grant-validity / eligibility / activity dashboard.

Library pins: `@stacks/connect@8.2.6` (latest, maintained by stx-labs), `@stacks/transactions@7.5.0`, `@noble/secp256k1@2.1.0` — matching signer-sidekick's own dependencies.
